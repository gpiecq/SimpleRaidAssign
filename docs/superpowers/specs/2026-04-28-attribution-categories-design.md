# Attribution Categories — Design Spec

- **Date:** 2026-04-28
- **Target version:** SimpleRaidAssign `1.2.0`
- **Status:** approved (brainstorm), pending implementation plan

## Goal

Add an optional category layer (e.g. `P1`, `P2`, `Adds`, `Burn`) inside a boss encounter so that attributions can be grouped, displayed and announced phase by phase. Existing flat raids continue to work untouched: categories are purely additive.

## Decisions (from brainstorming)

1. **Categories are optional.** A boss may have zero categories (legacy behaviour). An attribution may have no category and is then displayed in an implicit "Uncategorized" section at the top of the list.
2. **Broadcast emits one chat message per category.** Title format `[Boss - CategoryName]`. The "Uncategorized" bucket emits a generic `[Boss] ...` message identical to today, only when it has selected attributions.
3. **Display: collapsible sections.** Each category renders as a foldable header (`▼ / ▶`) with its attributions indented underneath. Each section has its own `+ Add Attribution` button. A separate `+ Add Category` button lives above the list.
4. **Re-categorisation lives in the right edit panel.** A `Category` dropdown next to the marker dropdown moves the selected attribution to another category (or to Uncategorized).
5. **Deleting a non-empty category moves its attributions to Uncategorized** (popup confirmation). Empty categories delete silently.
6. **Category creation uses a preset dropdown.** `P1, P2, P3, P4, Pull, Adds, Burn, Transition, Heroism, Custom...`. `Custom...` opens a free-text popup.
7. **Announce filter is hierarchical: master + per-category + per-row checkboxes.** Tri-state behaviour on the master and category checkboxes.
8. **Ergonomic details:**
   - Rename: double-click on the category label → inline `EditBox`. The `✎` button does the same for discoverability.
   - Reorder: `▲ ▼` arrows on the category header (categories) and existing arrows on the attribution row (intra-category, skipping foreign categories).
   - Fold state persisted in `SimpleRaidAssignCharDB.uiFold[raidKey..":"..encKey][catId]`, defaulting to expanded.
9. **Data-model approach: flat (`categoryId` on attribution).** `enc.order` stays the canonical global order; categories are an additional grouping layer. Selected over the nested approach because it preserves round-trip safety with v1.1 clients.

## Data model

Two new fields on the encounter, one optional field on the attribution.

```lua
encounter = {
  -- unchanged:
  name, instance, updated, updatedBy,
  attributions = {
    [attribId] = {
      marker, players, context, note,
      categoryId = nil | catId,        -- NEW. nil = Uncategorized
    },
  },
  order = { attribId1, attribId2, ... }, -- unchanged: global display order

  -- new:
  categories = {
    [catId] = { name = "P1", updated, updatedBy }
  },
  categoryOrder = { catId1, catId2, ... },
}
```

### Identifiers

`catId` is generated by the same `NewId("cat")` helper used for `enc_*` and `attrib_*`. Stable across renames, never reused.

### Invariants

- `categoryId = nil` ⇒ attribution renders in the implicit `Uncategorized` section at the top of the list.
- `categoryId = catId` with `enc.categories[catId] == nil` ⇒ UI treats the attribution as Uncategorized for rendering, but the field is **not** auto-corrected in the DB. This is a defensive read-side fallback for races where a comms push removes a category before the matching attribution update arrives.
- `enc.order` is the only source of truth for the relative order of attributions both within a category and within Uncategorized.
- `enc.categoryOrder` is the only source of truth for the order of category headers.

### Migration

`Core.lua` already runs a non-destructive merge against `DEFAULTS` on load. We add `categories = {}` and `categoryOrder = {}` to the encounter template inside `DEFAULTS` for new raids, and a lazy initialiser inside `Attributions:GetEncounter` (or a one-shot `NormalizeEncounter(raidKey, encKey)`) so legacy encounters lazily acquire `categories = {}` and `categoryOrder = {}` the first time they are read.

No mutation of existing attributions: `categoryId` defaults to `nil`, which means every legacy attribution shows up in Uncategorized and the boss looks identical to v1.1 until the user creates a category.

## UI changes (UI.lua)

### Layout

```
┌──────────────────────────────────────┐
│ Attributions - <BossName>            │  unchanged header
├──────────────────────────────────────┤
│ [☐] Select all   [+ Add Category]    │  master row + new button
├──────────────────────────────────────┤
│ ─ Uncategorized ──────────────────── │  shown only if non-empty
│ [☐] {rt8} Tanks: Greg, Pim           │
│                                      │
│ ▼ [☐] P1   ▲ ▼  ✎  🗑   [+ Attrib]   │  category header, expanded
│   [☐] {rt7} Adds: Bob                │
│   [☐] Heals: Carla, Diane            │
│                                      │
│ ▶ [☐] P2   ▲ ▼  ✎  🗑   [+ Attrib]   │  category header, collapsed
└──────────────────────────────────────┘
[+ Add Attribution]                       global, creates Uncategorized attrib
[Announce] [Preview]                      unchanged
```

### Category header row

A new builder `BuildCategoryHeaderRow(parent)` produces:

| Element | Behaviour |
|---|---|
| `▼ / ▶` chevron | Toggle fold state. Click anywhere on the chevron flips the state and persists it in CharDB. |
| `[☐]` checkbox (tri-state) | Reflects whether all/some/none of the category's attributions are selected for announce. Click toggles all of them at once. TBC client doesn't support a native mixed state, so the partial state is rendered with a dimmed checked texture. |
| Category name label | Static text. Double-click swaps it with an inline `EditBox`. `Enter` commits via `RenameCategory`; `Escape` or focus loss without `Enter` cancels. |
| `▲ ▼` arrows | Reorder this category in `categoryOrder`. Greyed when at the boundary. |
| `✎` button | Opens the same inline edit mode as double-click. |
| `🗑` button | Empty category → delete silently. Non-empty → `StaticPopup_Show("SRA_DELETE_CATEGORY")`. |
| `[+ Attrib]` button | Add an attribution with this `categoryId`, auto-select it, auto-expand the category. |

### Uncategorized header row

Same builder, simpler variant: only the chevron and the tri-state checkbox are interactive. No name, no rename, no delete, no reorder, no `+ Attrib`. It is hidden entirely when there are no Uncategorized attributions.

### Attribution row

Unchanged structure (`BuildAttribRow`, `UI.lua:1419`). Intra-category arrows reuse the existing per-row arrow controls; the underlying `MoveAttribution` is updated to skip rows of a different `categoryId` when looking for a swap partner (see Attributions API).

A 20px left indent is applied when the attribution belongs to a category (i.e. `categoryId ~= nil`); Uncategorized rows keep the current 0px indent.

### Refresh logic

`RefreshAttribList` (`UI.lua:1447`) is rewritten to:

1. Hide all rows (header + attrib).
2. Walk `enc.order` once to bucket attribIds by `categoryId` (`buckets[nil]` and `buckets[catId]`).
3. Render the Uncategorized header + its attribs (only if `#buckets[nil] > 0`).
4. For each `catId` in `enc.categoryOrder`, render the category header. If not folded, render its attribs (filtered from `buckets[catId]`, preserving their order from `enc.order`).
5. Sync the master checkbox tri-state.

Row pooling stays the same (`panel.rows`), but two pools are kept: `panel.attribRows` and `panel.headerRows` (because their geometry differs).

### `+ Add Category` flow

A new dropdown is wired with the preset list. Selecting a preset directly calls `Attributions:AddCategory(raidKey, encKey, presetName)`. Selecting `Custom...` opens a new `StaticPopup` `SRA_NEW_CATEGORY` with a single text field, max length 32, OK/Cancel.

### `Category` dropdown in the right edit panel

A new dropdown placed between the marker dropdown and the context input:

- `(none)` → `SetAttributionCategory(..., nil)`.
- One entry per `catId` in `enc.categoryOrder`, label = category name.
- Separator.
- `+ New Category...` → opens the same preset dropdown, on selection the new category is created and the attribution is reassigned to it.

The dropdown re-renders whenever `currentAttribKey` changes.

## Attributions API additions

In `Attributions.lua`:

```lua
-- Categories
function Attributions:AddCategory(raidKey, encKey, name) -> catId
function Attributions:RenameCategory(raidKey, encKey, catId, newName)
function Attributions:DeleteCategory(raidKey, encKey, catId)   -- moves attribs to Uncategorized
function Attributions:MoveCategory(raidKey, encKey, catId, delta)
function Attributions:IterateCategories(raidKey, encKey)        -- yields catId, category in categoryOrder
function Attributions:GetCategory(raidKey, encKey, catId)

-- Attribution placement
function Attributions:SetAttributionCategory(raidKey, encKey, attribId, catId | nil)
```

`MoveAttribution` is updated: instead of swapping with the immediate neighbour in `enc.order`, it scans for the closest neighbour with the same `categoryId` and swaps with that one. If no such neighbour exists in the requested direction, the call is a no-op.

`AddAttribution` accepts an optional `categoryId` argument. If omitted, defaults to `nil` (Uncategorized) — preserves the current call sites.

All new mutators fire `DATA_UPDATED` and call `Raids:Touch(raidKey)`. No additional granular callbacks are introduced — the existing `DATA_UPDATED`-driven full refresh is sufficient for the initial UI.

## Broadcast (`Broadcast.lua`)

The public API is unchanged: `BuildMessages(raidKey, encKey, filter)` still returns `{ msg1, msg2, ... }`. The internal logic is split:

```lua
function Broadcast:BuildMessages(raidKey, encKey, filter)
  local enc = ...
  local title = string.format(L.titleFmt, enc.name or "?")
  local out = {}

  -- Uncategorized first
  local segs = self:BuildSegments(raidKey, encKey, filter, "uncategorized")
  if #segs > 0 then
    for _, m in ipairs(ChunkMessages(title, segs)) do out[#out+1] = m end
  end

  -- One block per category, in categoryOrder
  for catId, category in NS.Attributions:IterateCategories(raidKey, encKey) do
    local catTitle = string.format("[%s - %s]", enc.name or "?", SanitizeChatText(category.name or ""))
    segs = self:BuildSegments(raidKey, encKey, filter, catId)
    if #segs > 0 then
      for _, m in ipairs(ChunkMessages(catTitle, segs)) do out[#out+1] = m end
    end
  end

  if #out == 0 then return { title .. " " .. L.noAttribs } end
  return out
end
```

`BuildSegments` gains a `categoryFilter` parameter (a catId, or the string `"uncategorized"`):

```lua
function Broadcast:BuildSegments(raidKey, encKey, filter, categoryFilter)
  local segments = {}
  for attribId, attrib in NS.Attributions:IterateAttributions(raidKey, encKey) do
    local match = (categoryFilter == "uncategorized" and attrib.categoryId == nil)
               or (categoryFilter == attrib.categoryId)
    if match and (not filter or filter[attribId]) then
      segments[#segments + 1] = FormatSegment(attrib)
    end
  end
  return segments
end
```

`Announce` and `Preview` are unchanged; they call the updated `BuildMessages`.

The 255-char chunking applies independently per block: each block is run through `ChunkMessages` with its own title, which prepends the title only to the **first** message of the block (current behaviour preserved). Overflow lines within the same block continue without a title prefix.

## Comms (`Comms.lua`)

No protocol bump (`SRA1` v2 stays). The serializer is generic and round-trips all keys, so the new fields propagate without any change to push/pull or merge logic. Last-write-wins on `updatedAt` continues to work because every category mutation also calls `Raids:Touch(raidKey)`.

### v1.1 ↔ v1.2 compatibility matrix

| Sender | Receiver | Behaviour |
|---|---|---|
| v1.2 | v1.2 | Full categories support. |
| v1.2 | v1.1 | New fields stored as unknown sub-tables in the DB. UI v1.1 shows the flat list as before. Edits round-trip without dropping the new fields. |
| v1.1 | v1.2 | No `categories` / `categoryOrder` keys in the payload → lazy initialiser creates them empty. All attribs show up as Uncategorized. |
| v1.1 | v1.1 | Unchanged. |

A v1.1 client editing an existing v1.2 attribution preserves its `categoryId` (only the fields v1.1 knows are overwritten). A v1.1 client adding a new attribution creates it without `categoryId` ⇒ Uncategorized on v1.2 — acceptable.

## Persistence

### SavedVariables

- `SimpleRaidAssignDB.raids[raidKey].encounters[encKey].categories` (account-wide, synced).
- `SimpleRaidAssignDB.raids[raidKey].encounters[encKey].categoryOrder` (account-wide, synced).
- `SimpleRaidAssignDB.raids[raidKey].encounters[encKey].attributions[attribId].categoryId` (account-wide, synced).

### CharDB

```lua
SimpleRaidAssignCharDB.uiFold = {
  ["<raidKey>:<encKey>"] = {
    ["<catId>"]            = true,    -- folded
    ["__uncategorized__"]  = false,   -- expanded (key absent ⇒ default expanded)
  }
}
```

Garbage-collected on `ADDON_LOADED`: drop entries whose `raidKey` or `encKey` no longer exists.

## Static popups

Two new entries in `Core.lua`'s static-popup table:

- `SRA_NEW_CATEGORY` — text input, OK/Cancel, `OnAccept` calls `Attributions:AddCategory(raidKey, encKey, name)` from popup data, max 32 chars, no validation against duplicate names.
- `SRA_DELETE_CATEGORY` — confirmation only, message `"Delete category '<name>' and move <N> attributions to Uncategorized?"`, OK/Cancel, `OnAccept` calls `Attributions:DeleteCategory(...)`.

Both popups use `PopupEditBox(self)` for TBC Anniversary compatibility (see `1.0.0` changelog entry).

## Counters and Raid Summary

Unchanged. `CountEncounters`, `CountAttributions` and `CountAssignedPlayers` ignore categories. Categories are not surfaced on the home screen.

## Out of scope

- Cross-encounter category templates ("apply Hyjal phases to Anetheron, Kaz'rogal..."): may be added later as a one-shot import from a per-instance preset list, not in this spec.
- Drag-and-drop reordering: explicitly rejected during brainstorming.
- Category-level notes (free text on the category itself): not requested, easy to add later by extending the category record.
- Localised category presets: presets stay English, consistent with the rest of the addon (`feedback_default_locale_english` memory).

## Implementation order (preview)

The implementation plan will be authored separately, but the natural order is:

1. Data model + `Attributions` API (categories CRUD, `SetAttributionCategory`, updated `MoveAttribution`).
2. Lazy migration in `Attributions:GetEncounter`.
3. `Broadcast.lua` per-category messages.
4. UI: category header row builder, refresh logic rewrite, fold state CharDB.
5. UI: `+ Add Category` dropdown, static popups, `Category` dropdown in the right panel.
6. UI: tri-state checkbox plumbing.
7. `.toc` + CHANGELOG + README + screenshots.

Each step is independently testable in-game without breaking the rest.
